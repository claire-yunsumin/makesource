**측정 원칙 (절대 규칙 1과 충돌 금지)**
- 계측은 전부 로컬: `tracing` 스팬 + `logs/perf.log`. 외부 텔레메트리·크래시 리포터 금지 유지.
- **측정 없이 머지 없음**: 성능 PR은 본문에 동일 시나리오 before/after 수치를 기록한다 (docs/08에 벤치 시나리오 추가, P0.2).
- 품질에 영향 줄 수 있는 변경(샘플러·스텝·정밀도·해상도)은 A/B 비교 뷰(T5.2, 동일 시드)로 품질 확인 후 적용하고, 판단 근거를 이 문서 부록 또는 docs/06에 남긴다.

**이미 잘 되어 있는 것 (건드리지 않는다)**
- 진행 전송: HTTP 폴링이 아닌 WebSocket push (`engine/client.rs:120-173`)
- 이미지 전달: base64 IPC가 아닌 파일 경로 + asset protocol(`convertFileSrc`) — 프론트·백 모두 확인
- `count` 1~4장은 개별 잡 반복이 아니라 latent 배치 1잡 (`engine/generation.rs:291`)
- 갤러리 keyset 페이지네이션 (`db/mod.rs:121-156`) + IntersectionObserver 무한 스크롤 + `loading="lazy"`
- 무거운 IO의 `spawn_blocking` 오프로드, jobId+이벤트 패턴(D-005), 생성 이벤트 리스너 1회 등록(`GenerateScreen.tsx:122-133`)

## P1. 즉효 최적화 (저위험·고효율, ~1일)

### P1.1 `[profile.release]` 추가 — 현재 부재 (`src-tauri/Cargo.toml`)
```toml
[profile.release]
lto = "thin"        # fat lto와 빌드시간/성능 비교 후 확정
codegen-units = 1
strip = true
```
- 기본값(lto off, codegen-units 16, strip 없음) 대비 바이너리 축소 + 런타임 개선. `panic = "abort"`는 Tauri 이벤트 루프 영향 확인 후 별도 판단.
- AC: `.dmg` 크기 before/after 기록, `cargo test` 그린.

### P1.2 SQLite 커넥션 튜닝 (`db/mod.rs:49-54`)
```rust
SqliteConnectOptions::new()
    .filename(db_path).create_if_missing(true)
    .journal_mode(SqliteJournalMode::Wal)
    .synchronous(SqliteSynchronous::Normal)
    .busy_timeout(Duration::from_secs(5))
    .foreign_keys(true)
```
- 현재 WAL·busy_timeout 미설정 → 생성 완료 시 DB 쓰기와 갤러리 읽기가 경합하면 지연·잠금 오류 여지. WAL은 읽기/쓰기 동시성의 표준 처방.
- AC: 마이그레이션 아님(PRAGMA) — 기존 DB 열림 확인, CRUD 테스트 그린.

### P1.3 보조 인덱스 (마이그레이션 `0002_perf_indices.sql`)
```sql
CREATE INDEX idx_gen_fav ON generations(favorite, created_at DESC) WHERE favorite = 1;
CREATE INDEX idx_gen_style ON generations(style_id, created_at DESC);
```
- `history_list`의 ♥/스타일 필터(`db/mod.rs:137-151`)가 현재 풀스캔. 텍스트 검색은 leading-wildcard `LIKE`라 인덱스 불가 — 히스토리 1만 장 이상에서 느려지면 FTS5 전환을 후속 검토(지금은 범위 외, 수치로 판단).
- AC: `EXPLAIN QUERY PLAN`으로 인덱스 사용 확인 테스트.

### P1.4 HTTP 클라이언트 재사용
- `run_generation`이 매회 `reqwest::Client::new()` (`generation.rs:263`), `track_progress`도 취소용 클라이언트 중복 생성 (`client.rs:135`). `Engine` state에 이미 있는 공유 클라이언트(`commands/engine.rs:11-15`)를 주입해 재사용.
- AC: 생성 경로에 `Client::new()` 0회 (기존 유닛 테스트 그린).

### P1.5 리소스 파싱 캐시
- 매 생성마다 디스크 재읽기+재파싱 중: presets, styles.json, settings.json (`generation.rs:229,257`), 번역 사전은 **키워드마다** 내장 사전까지 재파싱 (`prompt/translate.rs:66-78`), 워크플로 JSON도 매회 파싱 (`prompt/workflow.rs:56`), profiles.toml도 학습 시작마다 (`training/profiles.rs:55`).
- 처방: 내장(`include_str!`) 리소스는 `OnceLock<serde_json::Value/구조체>`로 1회 파싱. 사용자 파일(presets/styles/settings/사용자 사전)은 mtime 기반 캐시 — 외부 편집 시나리오(프리셋 파일 공유)가 있으므로 mtime 확인은 유지.
- AC: 파싱 캐시 유닛 테스트 (mtime 변경 시 재로드 포함). 단일 생성 지연에는 소폭이지만 A/B 동시 2잡·연타 시 누적 제거.

### P1.6 `gen://progress` 코얼레싱 (`commands/generate.rs:107`)
- 현재 샘플러 스텝마다 IPC 이벤트 → 프론트 Zustand `set` → `ResultGrid` 전체 재렌더 (`genSession.ts:71-74`). 학습 경로는 DB만 1% 스로틀하고 이벤트는 미스로틀 (`commands/training.rs:398-420`).
- 처방: 진행률 변화 ≥1% 또는 100ms 경과 시에만 emit (마지막 값은 반드시 전달). 페이로드 구조 불변 — 계약 변경 아님. `train://progress`에도 동일 적용.
- AC: 28스텝 생성에서 이벤트 수 28회 → ≤15회, 진행 바 체감 동일.

### P1.7 Python 스크립트 조건부 기록

## P2. 썸네일 파이프라인 (갤러리 체감 최대, ~1일)

**문제**: 썸네일이 아예 생성되지 않는다 — `thumb_path = image_path` (`engine/generation.rs:366`, 주석 "썸네일 생성은 갤러리(M3)에서"가 미이행). 갤러리 масonry가 원본 1024² PNG(장당 1.5~3MB)를 그대로 디코드 → K4 병목의 원인.

### P2.1 생성 시 썸네일 저장
- 파일 이동 직후 `spawn_blocking`에서 `image` crate(이미 의존성, webp 피처 활성)로 최장변 512px WebP(품질 ~80) 생성 → `outputs/YYYY-MM/{uuid}.thumb.webp`, `thumb_path`에 상대 경로 저장. 장당 예상 30~80KB (원본 대비 ~97% 감소).
- `gen://done`을 썸네일 완료 전에 emit(생성 체감 지연 금지), 썸네일은 후처리. 갤러리는 이미 `thumbPath || imagePath` 폴백이므로(`GalleryScreen.tsx:255`) 경합 안전.
- AC: 새 생성 레코드의 thumb_path ≠ image_path, 썸네일 크기 ≤ 100KB, 유닛 테스트(리사이즈 비율·경로 규약).

### P2.2 기존 레코드 백필
- 앱 시작 후 idle에 백그라운드 잡: `thumb_path == image_path`인 행을 낡은 순으로 스로틀(예: 초당 4장) 처리. 진행 이벤트 불필요(무소음), 실패 행은 skip.
- AC: 시드 1,000장 DB에서 백필 완주, 갤러리 스크롤 메모리 before/after 기록.

### P2.3 `<img decoding="async">` 부여 (갤러리·ResultGrid) — 메인 스레드 디코드 제거. 1줄.

## P4. 학습 속도 — kohya 인자 현대화 (~1.5일 + 실기기 검증)

**문제**: `build_kohya_args`(`training/runner.rs:181-230`)가 최소 인자만 구성 — fp32 고정(`--mixed_precision no`, runner.rs:211), latent/텍스트 인코더 캐시 미사용, SDPA 미지정. 에폭마다 VAE 인코딩과 텍스트 인코더 forward를 전부 반복하는 상태.

적용 순서 (각각 독립 A/B, 30장 fast 1에폭 시간으로 측정):

### P4.1 `--cache_latents --cache_latents_to_disk`
- 학습 이미지 VAE 인코딩을 1회로 — 에폭 수(4~12)만큼 반복 제거. 단독으로 가장 확실한 이득. 부작용 조건(랜덤 크롭·flip augment 미사용) 현재 구성과 충돌 없음 확인.

### P4.2 `--network_train_unet_only --cache_text_encoder_outputs`
- SDXL LoRA의 kohya 권장 조합: 텍스트 인코더 2개를 학습에서 제외하고 출력 캐시 → 스텝당 연산·메모리 대폭 감소. 스타일 LoRA(트리거워드 방식)에 적합 — 트리거는 캡션에 있으므로 TE 캐시와 양립. caption shuffle/dropout 미사용 확인 필요.
- 품질 영향 가능 → fast/standard 프로파일 A/B 샘플 비교 후 확정.

### P4.3 `--sdpa`
- PyTorch 2 scaled-dot-product attention. MPS 지원, xformers(CUDA 전용) 불필요. 저위험.

### P4.4 mixed precision bf16 (조건부)
- `no` 하드코딩을 "기기·PyTorch 버전 감지 후 bf16 시도, 실패 시 no 폴백"으로. MPS의 fp16 불안정 사유(주석)와 달리 bf16은 별개 — 단 accelerate의 MPS bf16 지원 여부가 관건이라 **실기기 실측 게이트** (불안정하면 채택 안 함, 결과를 D-xxx로).

### P4.5 보조 인자
- `--persistent_data_loader_workers --max_data_loader_n_workers 2` — 에폭 간 워커 재기동 제거.
- `--gradient_checkpointing`은 속도 아닌 메모리 처방 — 16GB에서 OOM 날 때만 프로파일 조건부.

### P4.6 fast 프로파일 해상도 768 검토
- 1024 → 768은 스텝 시간 약 절반. "빠름"의 목적(초안 확인)에 부합하는지 A/B 품질 확인 후 결정. 채택 시 `profiles.toml`의 `estimate_ko` 갱신 + 04 문구 톤 유지.

- **파서 주의**: latent 캐싱 도입 시 캐싱 tqdm 바가 진행률로 오인되지 않는지 — 현재 파서가 `steps` 접두 라인만 진행으로 취급(`training/parser.rs:101-106`)하므로 안전하지만, 회귀 테스트에 캐싱 로그 샘플 추가.
- 기대 종합: K5 30~60분 → **15~25분** (P4.1+P4.2 합산이 지배적).
