# 결정기록 (Decision Log)

버전: v0.1 | 작성일: 2026-07-05 | 상태: 운영 중

프로젝트의 주요 기술·운영 결정을 기록한다. 초기 결정(D-001~D-006)은 소급 기록.

**기록 규칙**
- 새 결정은 맨 아래에 D-번호를 이어서 추가하고, 번복 시 원래 항목을 지우지 말고 상태를 `대체됨 → D-xxx`로 바꾼다
- "결정"은 되돌리기 비용이 큰 선택(스택, 라이선스, 계약, 운영 정책)만 기록한다. 구현 세부는 PR 설명으로 충분

## D-001. 앱 셸: Tauri 2 + ComfyUI 별도 프로세스

- **날짜**: 2026-07-05 | **상태**: 확정
- **결정**: SwiftUI+Core ML 대신 Tauri 2(React/TS + Rust)를 앱 셸로, 추론은 ComfyUI를 별도 프로세스로 기동해 HTTP/WS로 통신한다.
- **이유**: 제로코스트 구축 기본안(docs/01). ComfyUI 생태계(워크플로·커스텀 노드)를 그대로 활용하면서 앱 코드는 GPL 전염 없이 유지 가능. 웹 프론트 역량 재사용.
- **영향**: ComfyUI는 저장소에 포함하지 않고 부트스트랩 설치기(T1.1)가 사용자 기기에 설치. PRD §8-1 해소.

## D-002. 저장소 public 전환 + main 브랜치 보호

- **날짜**: 2026-07-05 | **상태**: 확정
- **결정**: 저장소를 public으로 전환하고, main에 `protect-main` 룰셋 적용 (직접 push 금지, PR 필수, CI 통과 후 squash merge만 허용).
- **이유**: GitHub 무료 플랜에서 브랜치 보호 룰셋은 public 저장소에서만 동작. 트렁크 기반 전략(CLAUDE.md)의 "main 항상 그린"을 강제하기 위해 필요.
- **영향**: 공개 저장소가 되었으므로 라이선스 명시가 필요해짐 → D-006.

## D-003. DB: sqlx + SQLite

- **날짜**: 2026-07-05 (T0.3, PR #3) | **상태**: 확정
- **결정**: 로컬 DB는 SQLite, Rust 접근 계층은 rusqlite가 아닌 sqlx(runtime-tokio, migrate) 채택.
- **이유**: 컴파일 타임 쿼리 검증(macros), 내장 마이그레이션, async 런타임(tokio)과의 자연스러운 결합. Tauri command가 async이므로 동기 API인 rusqlite는 spawn_blocking 래핑이 반복적으로 필요.
- **영향**: 스키마 변경은 `migrations/`에 마이그레이션 파일 추가로만 수행. 스키마 정의는 TAD §4가 기준.

## D-004. 앱 데이터 경로: `~/Library/Application Support/LocalBrush/`

- **날짜**: 2026-07-05 (T0.3 fix, PR #4) | **상태**: 확정
- **결정**: 앱 데이터 루트를 TAD §3에 정의된 `LocalBrush/` 단일 폴더로 통일. 히스토리 이미지 경로는 이 루트 기준 상대 경로로 저장.
- **이유**: 초기 구현이 TAD와 다른 경로를 사용해 정합성 수정. 사용자가 저장 위치를 옮겨도 DB 레코드가 깨지지 않아야 함.

## D-005. long-running 작업: jobId + Tauri event 패턴

- **날짜**: 2026-07-05 (T1.1~) | **상태**: 확정
- **결정**: 오래 걸리는 command(부트스트랩, 생성, 학습)는 jobId만 즉시 반환하고 진행 상황은 Tauri event(`bootstrap://`, `gen://`, `train://`)로 push한다. 블로킹 command 금지.
- **이유**: UI 프리즈 방지, 취소·재시작 제어 지점 확보. command 시그니처는 TAD §5가 단일 기준.

## D-006. 라이선스: MIT

- **날짜**: 2026-07-05 | **상태**: 확정
- **결정**: 저장소 코드를 MIT 라이선스로 공개. 저작권자는 Claire Yun.
- **이유**:
  - ComfyUI(GPL-3.0)는 저장소에 포함하지 않고 사용자 기기에 별도 설치된 독립 프로세스와 통신만 하므로(D-001) GPL 전염 없음 — 라이선스 자유 선택 가능
  - PRD §8-4 "향후 유료화 가능성"에 열려 있으려면 허용적(permissive) 라이선스가 적합. 저작권자가 단독이므로 향후 조건 변경도 자유로움
  - 개인 공개 저장소의 사실상 표준. Apache-2.0의 특허 조항은 현 단계에서 과함
- **영향**: 새 의존성은 MIT/Apache/BSD 우선, GPL 추가 금지 원칙(CLAUDE.md 절대 규칙 2) 유지. PRD §8-2 해소.

## D-007. 기본 프리셋 축: 용도(이미지 타입) 기반 6종

- **날짜**: 2026-07-06 | **상태**: 확정
- **결정**: `presets.default.json`의 기본 프리셋을 그림체 기반 6종(플랫 벡터/수채화/라인 아트 등)에서 **용도 기반 6종**(기본/동화같은/인포그래픽/아이콘/3D 오브젝트/배너 — PRD F-2.1 예시 그대로)으로 교체한다.
- **이유**: 제품 설계상 두 축은 직교해야 한다 — **스타일 = 브랜드 톤·그림체**(에센스/LoRA가 담당, PRD Flow A의 "플랫 일러스트" 예시), **이미지 타입 = 용도**(인포그래픽/아이콘/배너). 초기 구현이 그림체를 이미지 타입 자리에 넣어 두 축이 겹쳤고, 사용자가 브랜드 스타일을 만들면 수채화·플랫벡터류 프리셋과 톤이 충돌하는 UX 문제가 생긴다. 토스트 원형("이미지 타입 + 개체")과도 용도 축이 부합.
- **영향**: 구 그림체 5종의 프롬프트 값은 폐기하지 않고 07 문서 부록 A에 "번들 스타일 후보"로 보존 (M4 에센스/번들 LoRA에서 재사용). `storybook`은 양쪽 목록에 공통이라 id 유지, 구 `3d-render` 값은 `3d-object`로 승계. 07 연구 문서는 새 6종 기준 v0.2로 재작성.

## D-008. 한→영 변환: 용어 사전 1차 + Argos Translate 2차 (Ollama 기각)

- **날짜**: 2026-07-06 (T2.3) | **상태**: 확정
- **결정**: 키워드 한→영 변환은 ① 내장+사용자 도메인 용어 사전(`dict.ko-en.json`) → ② Argos Translate(venv 서브프로세스, `python/translate.py`) → ③ 실패 시 원문 + 사용자 경고 순서로 처리한다 (TAD §4). Ollama 경량 LLM 경로는 기각.
- **이유**:
  - 사전: 짧은 명사 위주 키워드는 사전이 가장 정확·즉시·오프라인. 사용자가 브랜드 고유 용어를 직접 추가 가능
  - Argos(MIT, CC0 모델): 완전 로컬·무료로 로컬 전용 원칙(절대 규칙 1)과 부합, 모델 수백 MB 수준
  - Ollama 기각: 별도 런타임 상주 + 수 GB 모델은 짧은 키워드 변환에 과함. 번역 품질도 NMT 대비 이점 불확실
- **영향**: Argos 실설치(venv pip + ko→en 모델)는 부트스트랩 확장(T2.3b)로 분리 — 그전까지 사전 미스 키워드는 원문 폴백으로 동작. PRD §8-3 해소.

## D-009. 데이터셋 빌더는 평평한 작업 폴더, kohya 폴더 규약은 학습 시작 시점에 생성

- **날짜**: 2026-07-06 (T6.2) | **상태**: 확정
- **결정**: T6.2(데이터셋 빌더 + 캡션 편집)는 이미지를 `datasets/{id}/img-NNN.ext` + `img-NNN.txt`(캡션)로 평평하게 저장한다. TAD §8이 명시한 kohya 규약 폴더(`training/{style}/dataset/img/{repeat}_{trigger}/`)로의 재구성은 T6.3(`training_start`)이 프로파일(반복수)과 트리거 단어(스타일 이름)를 알게 되는 시점에 수행한다.
- **이유**: T6.2 시점에는 아직 스타일이 없다(스타일 자동 등록은 T6.4) — 트리거 단어와 반복수(profiles.toml, 프로파일별로 다를 수 있음)를 T6.2가 알 방법이 없다. 캡션 테이블의 "일괄 찾아바꾸기"가 트리거 단어를 넣는 자연스러운 지점이므로, 폴더 이름에 넣을 트리거는 이후 단계(③ 프로파일 선택 또는 학습 시작 직전)에 확정하는 편이 데이터 흐름과 맞는다.
- **영향**: `dataset_create`/`caption_dataset`/`dataset_save_captions`(T6.2)는 kohya 폴더 구조를 몰라도 된다. `training_start`(T6.3)는 이 평평한 데이터셋을 `img/{repeat}_{trigger}/`로 복사(또는 심링크)해 kohya에 넘겨야 한다 — T6.3 구현 시 반드시 반영.

## D-010. i18n은 v0.5에서 인프라만, 전체 번역은 v1.0

- **날짜**: 2026-07-06 (T7.2) | **상태**: 확정 (사용자 승인)
- **결정**: v0.5(내부 베타)에서는 i18n 인프라(언어 설정 ko/en — settings.json `language`, 경량 사전 `src/lib/i18n.ts`, `useT()` 훅)와 공용 컴포넌트·내비게이션의 en 번역만 구현한다. 화면별 문자열(약 740곳)과 백엔드 AppError 한국어 메시지의 en 번역은 v1.0 지역화에서 수행한다.
- **이유**: PRD 비기능 요구사항이 지역화를 v1.0 항목으로 명시("지역화 | v1.0 한국어/영어"). 백로그 T7.2의 "i18n ko/en"과 어긋나는 부분을 사용자에게 확인 — 인프라만 선반영하고 전체 번역은 미루기로 결정. T7.3(문구 정비)이 ko 문구를 손볼 예정이라 지금 번역하면 이중 작업이 되는 것도 근거.
- **영향**: en 선택 시 사전에 없는 화면은 ko로 노출되며 설정 화면에 그 사실을 고지한다. v1.0 지역화 시 화면별 문자열을 `i18n.ts` 사전(또는 화면별 분할 사전)으로 옮기고, 백엔드 에러 메시지 처리 방식(코드 기반 프론트 번역 vs 백엔드 locale 전달)을 그때 결정한다.

## D-011. 릴리스 파이프라인은 GitHub Releases(무서명)만 우선 활성화, Homebrew cask는 초안만

- **날짜**: 2026-07-06 (T7.4) | **상태**: 확정
- **결정**: `vX.Y.Z` 태그 push 시 `.github/workflows/release.yml`(tauri-apps/tauri-action)이 서명·공증 없이 `.dmg`를 빌드해 GitHub Release 초안(draft)으로 올린다. Homebrew cask는 `packaging/homebrew/localbrush.rb`에 초안만 작성해두고, 실제 `brew install --cask`로 배포하려면 필요한 별도 tap 저장소(예: `claire-yunsumin/homebrew-localbrush`)는 아직 만들지 않았다.
- **이유**: 새 GitHub 저장소 생성은 외부에 노출되는 인프라 변경이라 세션 내에서 임의로 만들지 않는 편이 안전하다 — PRD가 명시한 배포 채널(GitHub Releases + Homebrew cask, Mac App Store 보류) 중 GitHub Releases는 이 저장소 안에서 완결되므로 우선 구현했다. 릴리스를 draft로 두는 이유는 무서명 배포에 필요한 README 안내(우클릭 → 열기)를 배포 전에 한 번 더 확인하고 수동으로 공개(publish)하기 위함.
- **영향**: 태그를 push해도 Release는 자동 공개되지 않는다 — GitHub에서 draft를 확인 후 수동으로 "Publish release"해야 사용자에게 보인다. Homebrew tap 저장소를 만들기로 결정하면 `packaging/homebrew/localbrush.rb`를 그 저장소로 옮기고 릴리스마다 version/sha256을 갱신하는 자동화(release.yml 확장 또는 별도 워크플로)를 추가해야 한다.
